# System Design Notes

A structured set of notes exploring System Design fundamentals — built while learning how large-scale systems are designed to be scalable, reliable, and maintainable.

System Design isn't about memorizing jargon — it's about understanding trade-offs. Every design choice (SQL vs NoSQL, sync vs async, consistency vs availability) is shaped by what the system actually needs to do. These notes break down that jargon into clear, practical concepts.

## 📂 Contents

The notes are organized  and cover:

- **Fundamentals** — What system design really means, and the mindset behind it
- **Application Types** — Data-intensive vs compute-intensive systems
- **Requirements** — Functional vs non-functional requirements (scalability, availability, reliability, performance, security, maintainability, observability)
- **DNS** — How domain names resolve to IP addresses (Root → TLD → Name Server)
- **APIs** — REST, SOAP, GraphQL, gRPC, WebSockets — and when to use each
- **Databases** — SQL vs NoSQL, schema design, joins, and relationship types
- **Caching** — Cache strategies (read-through, write-through, write-around, write-back) and eviction policies (LRU, LFU, FIFO, MRU, LIFO)
- **Load Balancing** — Algorithms (round robin, least connections, weighted, IP hash, geo-based) and health checks
- **Distributed Databases** — Replication (single/multi-leader, leaderless), partitioning/sharding, and the CAP theorem
- **Message Queues** — Sync vs async processing, FIFO, priority queues, pub/sub, poison & duplicate messages
- **Fault Tolerance** — Designing for inevitable hardware, software, and human failures
- **Monitoring & Observability** — API-level and machine-level metrics etc....


## 🎯 Purpose

This repo serves as a personal reference and revision resource for system design concepts — useful for interview prep, architecture discussions, and building a stronger foundation as an AI/ML engineer who also understands how scalable systems are built.
